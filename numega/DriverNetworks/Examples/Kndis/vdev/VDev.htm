VDev NDIS Miniport Driver

[This is preliminary documentation and subject to change.]

Summary

Vdev demonstrates the use of 2 advanced features in the NDIS miniport framework:

·         Exposing WDM device objects

·         C++ Exception Handling in the kernel mode driver

Exposing WDM Device

VDev exposes a WDM device object named MyNdisDevice. This device object can be accessed by user mode programs via Win32 CreateFile(\\\\.\\MyNdisDevice, ... etc.) as a normal I/O device.

Underlying NDIS 5 services are NdisMRegisterDevice and NdisMDeregisterDevice. In the adapter's Initialize() handler, the driver creates the device object using NdisMRegisterDevice. In Halt(), the driver destroys it using NdisMDeregisterDevice. The static member function IrpDispatch() handles IRP processing. You can use the DriverWorks classes (such as KIrp) to simplify the processing of the I/O request.

Note that NdisMRegisterDevice  is not available on Win98 systems. Vdev can be conditionally compiled for Win98 targets when the preprocessor variable VDEV_WIN2K is set to 0. In this case, the driver uses IoCreateDevice to expose the device object.

C++ Exception Handling

C++ Exception Handling code is activated when preprocessor symbol VDEV_EH_DEMO is set to 1.  In Microsoft Visual Studio 6.0 IDE, choose configuration “Win32 NDIS 5 EH Demo Checked” to build the driver with EH code included. Method VdevAdapter::DoSomethingBad() demonstrates various cases of using C++ exception syntax to catch both kernel-raised and user-defined exceptions.

Building the Driver

Open Vdev.dsp in MSVC 6.0 IDE, select the “Debug” configuration and build it. The build creates files Vdev.sys and Vdev.nms. You will need DriverNetworks and the Microsoft Windows DDK installed to successfully build the driver.

Installing the Driver

You can use either Network Connection Manager or (better!) DriverStudio EzDrvInstaller utility (in the \Tools folder) to install the driver. Navigate to the .inf file and install it.

Tools

Use Driver Monitor Utility to view debug traces produced by the driver. If necessary, use SoftICE to step into and debug the driver code.
