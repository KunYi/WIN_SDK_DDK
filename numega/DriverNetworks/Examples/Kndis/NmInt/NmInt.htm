NmInt NDIS Intermediate Driver

[This is preliminary documentation and subject to change.]

Summary

This directory contains NDIS Intermediate Driver project files for NmInt. This sample demonstrates a generic NDIS Intermediate Driver.

An NDIS Intermediate Driver differs from a filter NDIS intermediate driver. Filters are easier to implement since the system takes care of managing binding relationships and there is always one virtual adapter per physical NIC in the system. Generic intermediate drivers are more flexible because they allow the driver to expose N virtual adapters to M physical NICs while having complicated binding arrangements. The driver could expose N virtual adapters for each physical NIC (N to 1.) Or the driver could expose one virtual adapter for all physical NICs in the system (1 to N). This flexibility comes with a price: the cost of the driver managing binding relationships.

In Windows 2000, a user mode component called a Notify Object is used to assist the driver with managing binding relationships. This is a COM server used by the Network Coinstaller and Network Configuration SubSystem to configure a Network Component and to display Property Pages in the Network Control Panel applet. A generic intermediate driver relies on the Notify Object to install virtual adapter instances which the driver exposes. Also, the Notify Object can alter existing bindings in the system if the intermediate driver wants to intercept all packets, such as disabling TCP/IP binding to a physical NIC.

In Windows NT, the OEMSETUP.INF file is used to install the driver. The relationships between the Physical NICs and Bindings are controlled using the Network Neighborhood application.

Building the Driver and Notify Object

First, build the DriverNetworks libraries. (You will need Microsoft Windows DDK installed to successfully build the library.) The Microsoft Visual Studio workspace is located in the “source” folder of the DriverNetworks installation folder. Open DNW.dsw in MSVC 6.0 IDE. Use the “Batch Build” command to build the “NDIS 5 Miniport Checked” and “NDIS 5 Protocol Checked” configurations of the KndisLib project. Also, build the Notify Object library. Open the Notify.dsw workspace which is located in the “source\Notify” folder of the DriverNetworks installation folder. Build the “Win32 Unicode Debug” configuration.

Next, build the NmInt driver. Open NmInt.dsw in MSVC 6.0 IDE, select the “NDIS 5 Miniport Checked” configuration of the NmInt project and build it. It creates files NmInt.sys and NmInt.nms.

Note:  Project “OID Handlers” is just a “dummy” project to aid in displaying OID handlers in the class browser of MSVC 6.0 IDE. This project cannot be built.

Finally, build the NmInt Notify Object. Using the same Microsoft Visual Studio workspace (NmInt.dsw), select the “Win32 Unicode Debug” configuration of the “IntMux” project and build it. It creates files NmIntNotify.dll and NmIntNotify.nms.

Installing the Driver

On Windows 2000 and Windows XP, the driver is installed as a Network Protocol Component Type. From the Local Area Connection Properties dialog, select the Install… button. Select the Protocol type of network component to install and click the Add… button. This launches the Select Network Protocol dialog. Click the Have Disk… button and browse to the netNmInt.inf file which is the .inf file for the protocol edge of the intermediate driver.

Note:  The miniport .inf file (netNmIntMP.inf) is not selected for install by the user.

At this point, the system will load the Notify Object. The Notify Object will prompt you to install virtual adapter instances for each physical NIC in the system. For each virtual adapter installed, the system will search for the .inf of the miniport part of the intermediate driver (netNmIntMP.inf)  This component is not digitally signed. If your system’s Driver Signing Options are set to Warn, meaning “warn the user when installing an unsigned driver,” the system will display a “Digital Signature Not Found” dialog four times for each virtual adapter installed. Select “Yes” for all of these dialogs to continue installation. Once the virtual adapter instances are installed, the existing bindings in the system for the physical NIC’s are disabled. Once the Notify Object processing is complete and the dialogs are dismissed, the NmInt driver will be loaded. All packets will now be intercepted by the intermediate driver.

For WinNT 4.0 an OEMSETUP.INF file is provided to simplify the installation process. Right-click the desktop Network Neighborhood icon. Then, select Properties. The network configuration application will run. On the network configuration application, select the Protocols tab. The network Protocols configuration page will be displayed. Select the Add button. Finally, the “Select Network Protocol” window will be displayed. At this point you are ready to install the protocol. Select the Have Disk option and perform the installation.

Tools

BoundsChecker for Drivers and SoftICE can be used to test some of the features of this driver. Specifically, the SoftICE “packet” command is useful for displaying and decoding NDIS packet buffers.

How It Works

The driver acts as a protocol driver on its lower edge for binding to real miniports in the system. The class NmIntBinding abstracts this protocol driver functionality. The driver also acts as a miniport driver on its upper edge for binding to real protocols in the system. The class NmIntAdapter abstracts this miniport driver functionality. Which real protocols and miniports the driver binds to and how many virtual adapters are exposed is controlled by the Notify Object dll. The driver must manage the internal bindings between NmIntBinding and NmIntAdapter instances. For example, when a received packet is indicated to our protocol edge by a real miniport in the system, the NmIntBinding instance must “choose” which NmIntAdapter instances to indicate the received packet up to real protocols. For this sample driver, the packet filter of the NmIntAdapter instances is compared with the destination address of the packet for deciding whether to indicate the packet. The underlying NIC is put into ‘promiscuous’ mode so that the virtual adapters will have a chance to see all packets. Also, when a real protocol sends a packet to our miniport edge, the NmIntAdapter instance must choose which underlying real miniport to send the packet by “choosing” a particular NmIntBinding instance.
